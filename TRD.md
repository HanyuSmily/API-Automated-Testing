# 开源电商项目 `mall` 全链路 API 自动化测试需求文档 (TRD)

---

## 1. 文档概述与测试目标

### 1.1 背景说明

当前 `mall` 项目的 API 自动化测试框架已实现底层引擎（pytest + requests）与 Mock 服务流转的闭环。然而，现有的 3 组基础用例（主流程 E2E、商品分页、异常查询）仅覆盖了系统的冰山一角。为了满足大厂生产级质量平台的要求，必须将测试用例库进行立体化扩充。

### 1.2 测试目标

* **多链路覆盖**：打通 `mall-portal`（前台商城）与 `mall-admin`（后台管理）的跨系统联动。
* **多维度覆盖**：从单接口的**边界值/异常值校验**，延伸至多接口串联的**状态机逆向全链路**。
* **数据双重校验（白盒思维）**：不仅断言接口 Response 的 Code，还要深度断言内部状态（如库存扣减、优惠券状态变更）。

---

## 2. 系统核心接口与业务拓扑

自动化测试框架将聚焦于以下四大核心模块的联动：

```
[UMS 会员模块] ──> [PMS 商品模块] ──> [SMS 营销模块] ──> [OMS 订单模块]
  (登录/鉴权)        (商品/库存)        (优惠券/满减)       (下单/履约/退款)

```

---

## 3. 全链路 E2E 业务场景设计

大厂面试中最看重**复杂业务场景的闭环设计**。以下规划三条最核心的端到端（E2E）测试链路：

### 链路 A：B2C 核心购买与履约正向全链路（跨系统联动）

* **场景流转**：
1. `[Admin端]` 发布新商品，设置初始库存为 100 件。
2. `[Portal端]` 会员登录，搜索该商品并进入详情页。
3. `[Portal端]` 添加商品至购物车，确认结算生成预订单。
4. `[Portal端]` 提交订单，生成订单号，此时单据状态为 `0`（待支付）。
5. `[Admin端]` 模拟支付回调，将订单状态变更为 `1`（待发货）。
6. `[Admin端]` 调用发货接口，输入物流单号，状态变更为 `2`（已发货）。
7. `[Portal端]` 会员确认收货，状态变更为 `3`（已完成）。


* **数据断言**：最终校验商品库存应精准变更为 99 件，且订单全生命周期状态机流转无误。

### 链路 B：SMS 营销领券核销全链路（高阶业务逻辑）

* **场景流转**：
1. `[Admin端]` 创建一张“满100减20”的商品优惠券，限定全网发放 10 张。
2. `[Portal端]` 会员 A 登录，调用领券接口。
3. `[Portal端]` 会员 A 将价值 120 元的商品加入购物车。
4. `[Portal端]` 提交订单，传入优惠券 ID 进行核销。


* **数据断言**：
* 断言订单应付金额（`payAmount`）应严格等于 $120 - 20 = 100$ 元。
* 调用“我的优惠券”接口，断言该券状态已变为“已使用”。



### 链路 C：超时未支付与逆向退款链路（异常与容错）

* **场景流转**：
1. `[Portal端]` 会员下单，锁定库存 1 件，单据进入待支付状态。
2. `[系统层]` 模拟超时逻辑（或直接调用取消订单接口）。
3. `[Admin端]` 后台触发订单关闭，状态变更为 `4`（已关闭）。


* **数据断言**：断言由于订单关闭，之前被锁定的商品库存必须**自动回滚释放**，库存恢复原值。

| 模块分区 | 核心接口 URL | 正向功能用例 | 边界值场景用例 | 异常/逆向阻断用例 |
| --- | --- | --- | --- | --- |
| **UMS (前台用户)** | `POST /sso/login` | 输入正确账号密码，成功换取 Token 并存储。 | 密码包含特殊字符、超长字符串登录校验。 | 1. 账号不存在<br>

<br>2. 密码错误<br>

<br>3. 连续错误 5 次触发账户锁定。 |
| **PMS (商品管理)** | `GET /pms/product/list` | 传入 `pageNum=1&pageSize=5`，正确返回 5 条数据。 | 1. `pageSize=0` 的容错处理<br>

<br>2. `pageSize=1000` 的大页数限制。 | 传入非法的参数类型（如 `pageNum=abc`），预期返回 400 或业务错误码。 |
| **OMS (购物车)** | `POST /oms/cart/add` | 商品首次加购，购物车新增记录；重复加购，数量累加。 | 1. 加购数量等于商品最大库存<br>

<br>2. 达到购物车单品上限（如 99 件）。 | 1. 加购已下架的商品<br>

<br>2. 加购库存为 0 的商品，预期阻断并报错。 |

<br>2. 领取已过期的优惠券。 |

## 5. 白盒化数据一致性与数据清理策略

### 5.1 数据双重校验（DB 级别断言）

在自动化框架中，不能仅依赖 `response.json()['code'] == 200`。在编写复杂的 E2E 脚本时，必须引入以下白盒断言：

* **库存一致性**：`Assert: 扣减后的真实库存 == 初始库存 - 订单商品数量`
* **财务一致性**：`Assert: 订单总金额 == 商品单价 * 数量 - 优惠券抵扣金额 + 运费`

### 5.2 高效的自动化造数与清理（Teardown 规约）

为了防止自动化测试脚本在 Mock 服务或测试数据库中留下大量脏数据，每个测试类/测试函数必须遵循以下生命周期：

```
[Setup 阶段] ──> [Execute 阶段] ──> [Assert 阶段] ──> [Teardown 阶段]
 动态插入测试账户    运行全链路测试用例    双重断言业务及数据    清空测试生成的订单/
 及特异性商品数据                                           恢复商品初始库存状态

```

## 6. 后续自动化脚本开发实施计划

你可以将这份需求文档作为“上下文”直接提供给 Trea，并按以下步骤命令其分批输出代码：

1. **第一阶段**：命令 Trea 根据第 4 节的接口测试矩阵，在 `mock_mall_server.py` 中补全所有缺失的边界和逆向接口逻辑（例如：支持返回库存不足、密码锁定、重复领券等状态码）。
2. **第二阶段**：命令 Trea 依据第 3 节的【链路 B】和【链路 C】，在 `testcases/` 目录下新建 `test_mall_coupon_e2e.py` 和 `test_mall_refund_e2e.py`，完善高阶业务链路。
3. **第三阶段**：利用 `pytest.mark` 标签对用例进行分类管理（如 `@pytest.mark.smoke` 代表冒烟主链路，`@pytest.mark.exception` 代表异常边界），从而支撑 AI Agent 能够通过更精细的指令选择性地跑不同的测试集。
